# 🎓 学生保研信息管理系统 (SPRIMS)

## 📖 项目介绍
**学生保研信息管理系统 (Student Postgraduate Recommendation Information Management System)** 是一款基于 Java Swing + MySQL 开发的桌面级应用程序。
本项目旨在为高校的“推免生（保研）”工作提供一个全流程、数字化的管理平台。系统严格划分了**学生、教师、管理员**三层角色权限，实现了从学生提交申请、上传佐证材料，到教师查阅初审、综合打分，再到管理员全局监控、政策管理的业务闭环。

## 🛠️ 环境配置
本项目为标准 Java SE 项目（未使用 Maven），依赖包已在本地提供。
* **开发语言**: Java (我使用JDK 17)
* **数据库**: MySQL 8.0 及以上
* **开发工具**: IntelliJ IDEA 
* **外部依赖 (Jar包)**:
    * `mysql-connector-j-8.x.x.jar` (MySQL 驱动)
    * `flatlaf-3.4.1.jar` (现代化扁平化 UI 主题)

## 🚀 运行步骤
1.  **克隆项目**到本地：
    ```bash
    git clone [https://github.com/你的用户名/你的仓库名.git](https://github.com/你的用户名/你的仓库名.git)
    ```
2.  **配置数据库**：
    * 在 MySQL 中新建数据库 hrmisss
    * 导入项目附带的 `.sql` 数据库脚本文件，生成所需的表结构及初始数据。
    * 打开 `src/com/hrmisss/util/DBUtil.java`，修改数据库连接的 URL、用户名和密码为你本地的配置。
3.  **导入依赖包**：
    * 在 IDEA 中，选中项目根目录下的 `.jar` 包文件，右键点击 `Add as Library...`。
4.  **运行程序**：
    * 找到并运行入口文件 `src/com/hrmisss/view/LoginFrame.java` 的 `main` 方法，即可启动系统。

## ✨ 功能说明
系统包含三大核心模块，对应三种不同的登录角色：

### 🧑‍🎓 1. 学生端
* **保研申请提交**：选择意向保研政策及方向，填写申请表。
* **佐证材料上传**：支持将本地证书/截图（如 PDF、PNG、MP4）上传至系统，实现物理文件存储与数据库路径的精准映射。
* **进度追踪**：实时查看个人保研申请的初审状态及教师给出的综合评分。

### 👨‍🏫 2. 教师端
* **待办列表处理**：分类查看所有处于“待初审”状态的学生申请。
* **材料查阅与审核**：一键调阅学生上传的本地佐证材料，核对真伪。
* **初审与打分**：填写初审意见，执行“通过”或“驳回”操作，并对学生的创新成果进行综合打分。

### 🛡️ 3. 管理员端
* **全局监控与强力干预**：拥有系统最高权限，可穿透查看所有状态的申请（包括教师已审/未审）；在特殊情况下，可通过特权按钮强制修改学生申请状态。
* **保研政策管理**：发布、修改、删除各学院的保研政策及加分细则。
* **信息查询与筛选**：支持按学号、学院、综合得分等多维度筛选学生，提供数据支撑。

## 💡 个人修改/扩展内容
在基础业务逻辑跑通后，本人对系统进行了深度的二次开发与重构，核心亮点如下：

1.  **UI 界面现代化大改版 (FlatLaf 注入)**
    * 彻底废弃了老旧的 Java Swing 默认外观，全局引入了 `FlatLaf` 扁平化主题。
    * **细节打磨**：通过 `UIManager` 深度定制了组件的圆角 (`Component.arc`)，重构了 `JTable` 渲染器，实现了表格边框的完美闭合与列宽的自适应按比例分配。
    * **交互优化**：为所有数据密集型表格增加了 Hover Tooltip (鼠标悬停提示) 机制，解决了长文本（如长政策名、超长学号）被截断显示的问题。
2.  **业务架构与角色权限重构**
    * **解决职责重叠**：重构了底层 DAO 层与 UI 层的查询逻辑，修复了初期“管理员与教师功能完全重叠”的逻辑缺陷。
    * **确立层级审批流**：将管理员端重新定位为“全局监控平台”，加入了“特权警告”拦截机制，确保了业务流程符合真实高校场景。
3.  **文件 I/O 与数据一致性修复**
    * 重写了 `MaterialUploadFrame` 的核心逻辑，解决了 Java `NIO Files.copy` 物理文件落盘成功但 MySQL 数据库字段映射失败的“脏数据” Bug。
    * 确立了“相对路径存储 + 绝对路径映射”的双规文件管理机制。
